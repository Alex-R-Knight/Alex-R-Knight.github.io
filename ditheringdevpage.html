Rendering - Dynamic Colour Quantisation and Dithering:

Initial Foundations:

I set out to create postprocessing to emulate retro-graphics through the use of colour quantisation and dithering.
I wanted to achieve this whilst preserving the fidelity of user interfaces like my terminals.

I started with postprocessing to downsample the source image, and quantise the colour.
I used Bayer noise to apply dithering to the quantisation values, smoothing the transition between quantised colours.
The quantisation number for each colour may be set individually, and changed during runtime for effects.

I experimented with applying different downsampling/quantisation to objects, using Unreal Engine's custom stencil buffer.
This let me increase the fidelity of elements such as user interfaces and objects that are currently bring interacted with.
This worked suitably, allowing UI elements to be presented in the same style as their surroundings but without harming their legibility.

Whilst experienced in writing rendering code from my Global Illumination Project, I prototyped the material entirely with the engine's node-based system.
Hence, I adapted my understanding, utilising material functions where possible to avoid repetition.

This presented some issues like a seeming lack of matrices within the editor, which I found solutions to such as the reading of a texture to provide the noise values used for dithering.

Improved Passing of Data to Postprocessing:

In my early work, I passed values by creating a dynamic material instance for each postprocess.
This took a convoluted process where on level-start the postprocess volume was found and it's materials were replaced with these instances.

That was messy, so I began using Material Parameter Collections.
These were much better to work with, passing data to materials with no need to create and manage several dynamic instances.

HLSL Usage in Materials:

With the node-based prototype working, I revisited it and rewrote large portions in HLSL to tidy up and optimise my most intensive postprocess.
This reduced computational cost of the material and made it far more readable than the previous node-spaghetti.

Applying Parameter Collections - Dynamic Vertex Snapping:

With the utility of material parameter collections, I experimented with some other effects.
Amongst these was emulating the "vertex snapping" of older hardware, scaling in its intensity based on the player's health.

Whenever player health changes, it is mapped into a vertex snapping value and passed to a MPC.
This is read by a material function affecting the world position offset of materials.

I can easily add or remove this function from any material and object I need.
Within the function, the camera-relative world position is divided, floored, and then multiplied by the scaled parameter.
The result of this is subtracted from the relative world position to give the offset applied to the vertex.

Optimising Dynamic Materials through Custom Primitive Data:

My early work changed mesh appearances - such as switching the emissive texture of a lightbulb - by replacing the material itself.
This poor approach was quickly replaced with dynamic material instances where a scalar could interpolate between emissive textures.

To further improve this, I began using custom primitive data to pass information to a shader without needing dynamic material instances.

This approach has worked wonderfully, and allows for the easy adjustment of parameters within my C++ code.


// Called during beginplay
UPrimitiveComponent* SignalPrimitive = Cast(SignalLightMesh);

// Called as required to change appearance
SignalPrimitive->SetCustomPrimitiveDataFloat(1, 1.0f);

Custom primitive data is well suited for working with instanced static meshes, where each instance may have its own custom primitive data.
This bypasses the inability to assign different materials or instances to mesh instances, which can be applied in all sorts of ways from object colour to more advanced effects.
This is important due to the use of instanced meshes to reduce draw calls and improve performance.

Overlay Material Usage:

I found I sometimes needed to apply visual effects to numerous objects at some point during runtime, but it would be inefficient to factor this into their different materials.
I began using overlay materials, which can be applied and removed during runtime.

These create an additional draw call and can be utilised for numerous visual effects - in my case applying a sheen to show an object is ready to be released.
By ensuring overlay materials are removed when they are no longer needed, any performance impact is negligible, whilst the materials of the base-objects remain consistently simple.

Overlay materials can also be managed in code, allowing me to control them through the component that grants an actor object carrying logic.


TArray MeshComponents;
GetOwner()->GetComponents(MeshComponents, true);

for (UStaticMeshComponent* MeshComponent: MeshComponents)
{
    MeshComponent->SetOverlayMaterial(OverlayMaterial);
}

Using this and custom primitive data has been wonderful for creating more visually dynamic objects that are simple to control within the code.
